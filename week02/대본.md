## 슬라이드: 1.1 React란?

**멘트**

> "**React**는 **사용자 인터페이스(UI)를 만들기 위한 자바스크립트 라이브러리**입니다. 웹뿐 아니라 모바일(React Native) 등으로도 확장되지만, 오늘은 **웹 브라우저 화면** 기준으로 설명할게요."

> "**개발사**는 **Facebook, 지금은 Meta**입니다. 오래 쓰이고, 회사 규모도 크고, **채용 시장에서도 자주 요구**되는 편이에요."

> "**특징** 네 가지를 외우기보다는 **느낌**만 잡으세요.  
> **컴포넌트 기반** — 아까 말한 레고 조각.  
> **가상 DOM** — 곧 자세히 말하지만, **진짜 화면을 매번 통째로 바꾸지 않고 똑똑하게 바꾸는 방식**이에요.  
> **단방향 데이터 흐름** — 데이터가 **한 방향으로만** 흐른다는 뜻인데, 오늘은 **부모가 자식에게 Props를 준다** 정도만 연결해 두면 됩니다.  
> **재사용 가능한 UI** — 같은 버튼 컴포넌트를 여러 페이지에서 쓴다, 이런 그림이에요."

**질문 받을 때**

- "프레임워크냐 라이브러리냐?" → "공식적으로는 **UI에 집중한 라이브러리**로 소개됩니다. 앱 전체를 다 규정하지 않고, **화면을 그리는 부분**에 가깝게 쓰는 경우가 많아요."

**전환**

> "그중 **가상 DOM**은 면접이나 글에서 자주 나오니까, 비전공자도 이해할 수 있게 한 번 더 풀어 볼게요."

## 슬라이드: 가상 DOM (Virtual DOM)이란?

**멘트**

> "**실제 DOM**이 뭘까요? 브라우저가 HTML을 읽어서 만든 **트리 구조의 객체**예요. 화면의 각 요소( div, button 등 )가 **객체로 연결**되어 있어요. 이걸 **직접** 아주 자주, 아주 세밀하게 바꾸면 **성능 부담**이 생길 수 있어요."

> "**가상 DOM**은 React가 **메모리 안에 들고 있는 가벼운 복사본** 같은 거예요. 진짜 화면이 아니라 **계산용 모형**에 가깝습니다."

> "**동작 순서**를 말로 해볼게요.  
> 1) **State 같은 게 바뀐다**  
> 2) React가 **새 가상 DOM**을 그려요  
> 3) **이전 가상 DOM과 비교(diff)** 해서 **달라진 부분**을 찾아요  
> 4) **그 부분만 실제 DOM에 반영**해요  

> 그래서 "매번 전체 페이지를 새로 그린다"보다 **효율적**인 경우가 많다는 **이미지**를 가지시면 됩니다. 알고리즘 세부까지는 초반에 몰라도 괜찮아요."

**비유 (선택)**

> "전체 벽지를 바꾸지 않고 **찍힌 자국만 칠한다**고 생각해도 됩니다."

**전환**

> "그럼 React를 쓰면 **실무적으로 뭐가 좋은지** 장점 슬라이드로 정리할게요."

## 슬라이드: 1.2 React의 장점

**멘트**

> "**컴포넌트 재사용** — 한 번 만든 걸 여러 곳에 쓰면 **일이 줄어요**.  
> **가상 DOM** — 아까 말한 것처럼 **바뀐 만큼만** 반영하려는 쪽으로 이해하시면 됩니다.  
> **풍부한 생태계** — 라우팅, 상태 관리, UI 키트 등 **주변 도구가 많아요**.  
> **커뮤니티** — 막히면 검색했을 때 **자료가 많다**는 건 학습과 업무 모두에서 큰 장점이에요."

**전환**

> "코드로 **바닐라 JS와 React**를 한 번 비교해 볼게요. 오늘 이후로는 'return 안에 HTML 비슷한 게 있다'에 익숙해지시면 됩니다."

## 슬라이드: 1.3 React vs Vanilla JavaScript

**멘트 (슬라이드 코드 기준)**

> "**왼쪽, 바닐라 JS**입니다. `document.createElement`로 버튼을 만들고, `textContent`로 글자를 넣고, `addEventListener`로 클릭을 달고, `appendChild`로 body에 붙입니다. **명령이 한 줄씩** 이어져요. 작은 예제는 괜찮은데, 화면이 커지면 **어디서 뭘 만졌는지** 추적이 어려워질 수 있어요."

> "**오른쪽, React**입니다. `Button`이라는 **함수**가 있고, **return** 안에 **JSX**로 버튼 모양을 씁니다. `onClick`에 함수를 넘기죠. **'이 컴포넌트의 화면은 이렇게 생겼다'**가 한눈에 모여 있어요."

> "초보 때 헷갈리는 포인트 하나: React 쪽 `onClick`은 **문자열이 아니라 중괄호 안의 자바스크립트 함수**예요. 이벤트 파트에서 다시 잡을게요."

**전환**

> "이제 **2부, JSX 문법**입니다. 여기서부터는 **문법 규칙**을 손에 익히는 시간이에요."

## 슬라이드: 2.1 JSX란?

**멘트**

> "**JSX**는 **JavaScript XML**이라는 뜻이에요. 자바스크립트에 **XML/HTML처럼 생긴 문법**을 섞어 쓰는 방식입니다."

> "내부적으로는 **React.createElement(...)** 를 여러 번 호출하는 걸로 변환됩니다. 그래서 문법적 설탕, **편하게 쓰기 위한 문법**이라고 소개되기도 해요."

> "**왜 className이에요?** HTML에서는 `class`인데, 자바스크립트에서 `class`는 **예약어**라서 다른 이름을 씁니다. React JSX에서는 **`className`** 입니다."

> "**왜 style은 객체예요?** JSX는 결국 **자바스크립트 표현식** 안에서 쓰이기 때문에, `style="color: red"` 같은 문자열 한 덩어리보다 **`{{ color: 'red' }}` 처럼 객체**로 주는 방식이 일반적이에요. 중괄호가 두 겹인 건 **바깥은 JSX용, 안쪽은 자바스크립트 객체**라서 그래요. 처음엔 어색해도 곧 익숙해져요."

**전환**

> "기본 문법 예제를 슬라이드로 같이 읽어 볼게요."

---

## 슬라이드: 2.2 JSX 기본 문법

**멘트 (코드 줄별)**

> "`const element = <h1>Hello, World!</h1>` — 변수에 **태그처럼 생긴 JSX**를 넣을 수 있어요."

> "`const name = "John";` 다음에 `<h1>Hello, {name}!</h1>` — **중괄호** 안은 **자바스크립트 구역**이에요. 변수 이름, 계산식, 함수 호출 결과를 넣을 수 있어요."

> "`<img src="..." alt="..." />` — 속성은 HTML과 비슷합니다. **alt**는 접근성 때문에 중요해요. 이미지가 안 보일 때 대체 텍스트입니다."

> "`<div className="container">` — 다시 한 번, **class가 아니라 className**입니다."

**데모 제안 (짧게)**

> "강사 화면에서 `name` 값을 바꾸고 저장하면 문구가 바뀌는 것만 보여 주셔도 좋아요."

**전환**

> "이제 **JSX 규칙**입니다. 여기서 에러가 많이 납니다."

## 슬라이드: 2.3 JSX 규칙 — 하나의 부모 요소

**멘트**

> "**규칙 1: 컴포넌트가 return하는 JSX는 보통 하나의 부모로 감싸야 해요.**"

> "슬라이드에 **잘못된 예**가 있어요. `return (` 다음에 `<h1>`과 `<p>`가 **나란히** 있으면, 형제만 두 개 있어서 **부모가 없는 것처럼** 되어 버려요. React는 이런 경우를 허용하지 않습니다."

> "**올바른 예**는 `<div>`로 **한 번 감싼 것**이에요. 이제 **부모 하나** 아래에 자식이 두 개 있으니 OK입니다."

> "나중에 **불필요한 div가 싫다**면 Fragment를 씁니다. 바로 다음 슬라이드예요."

**전환**

> "Fragment와 자기 닫는 태그를 볼게요."

## 슬라이드: JSX 규칙 (계속) — Fragment, 자기 닫는 태그

**멘트**

> "**Fragment**는 `<> ... </>` 입니다. **DOM에는 추가 노드 없이**, 논리적으로만 **한 덩어리로 묶을 때** 씁니다. 스타일링할 래퍼가 꼭 필요 없을 때 유용해요."

> "**자기 닫는 태그** — `<img />`, `<br />`, `<input />` 처럼 **내용이 비어 있는 태그**는 XML 규칙에 맞게 **`/`** 로 닫아줍니다. HTML에서는 관대한 경우도 있지만, JSX에서는 **닫아 주는 습관**이 안전해요."

**전환**

> "JSX가 어느 정도 보이셨으면, **3부, 컴포넌트**로 넘어갑니다. **레고 블록을 함수로 만드는 법**이에요."

## 슬라이드: 3.1 컴포넌트란?

**멘트**

> "**컴포넌트**는 **UI를 독립적이고 재사용 가능한 조각으로 나눈 것**입니다."

> "**종류**는 전통적으로 **클래스형**도 있었는데, 지금은 **함수형 컴포넌트**가 **권장**이에요. 새로 배울 때는 **함수형만** 마스터하시면 됩니다. 클래스형은 **레거시 코드**를 읽을 때 가끔 보는 정도로 알아 두면 돼요."

**전환**

> "함수형 컴포넌트 문법을 슬라이드 예제로 보겠습니다."

---

## 슬라이드: 3.2 함수형 컴포넌트

**멘트**

> "`function Welcome() { return <h1>...</h1>; }` — **일반 함수**로 컴포넌트를 정의합니다. 이름은 **대문자로 시작**해야 React가 **컴포넌트**로 인식해요. 소문자로 시작하면 HTML 태그로 착각할 수 있어요."

> "`const Welcome = () => { return ... }` — **화살표 함수**로도 똑같이 씁니다. 팀 컨벤션에 따라 선택해요."

> "`const Welcome = () => <h1>...</h1>` — return할 게 **한 줄**이면 **중괄호와 return을 생략**할 수 있어요."

**전환**

> "이제 **만든 컴포넌트를 어떻게 쓰는지**예요. HTML 태그처럼 **`<Welcome />`** 이라고 씁니다."

---

## 슬라이드: 3.3 컴포넌트 사용

**멘트**

> "`App` 안에서 `<Welcome />`을 **두 번** 썼죠. **같은 컴포넌트를 여러 번** 배치할 수 있어요. 아직은 똑같은 글자만 나오지만, 다음 파트 **Props**를 배우면 **각각 다른 이름**을 넣을 수 있어요."

> "여기서 포인트: **컴포넌트 이름은 항상 대문자 시작**, 사용할 때도 **`<Welcome />`** 처럼 대문자입니다."

**전환**

> "데이터를 **밖에서 안으로** 넘기는 방법이 **Props**입니다. **4부**로 갑니다."

---

## 슬라이드: 4. Props (60분) — 섹션 구분

**멘트**

> "Props는 **부모에서 자식으로 값을 넘기는 통로**예요. 여기서 **읽기 전용**이라는 말이 계속 나옵니다. **자식이 Props를 마음대로 바꾸지 않는다**는 규칙을 이해하는 게 중요해요."

---

## 슬라이드: Props vs State, 쉽게 구분하는 법

**멘트**

> "**Props**는 **부모가 건네준 데이터**예요. **받기만** 하고, 그 컴포넌트 안에서 **임의로 바꾸지 않습니다.** 예: 제목 문자열, 카드에 표시할 이름."

> "**State**는 **이 컴포넌트가 스스로 들고 있는 데이터**예요. **버튼 클릭, 입력** 같은 일로 **바뀔 수 있어요.** 예: 카운트 숫자, 체크박스 켜짐 여부."

> "**헷갈릴 때 질문 하나**: **이 값이 사용자 동작으로 바뀌나요?** — **바뀐다**면 대개 **State**, **안 바뀌고 부모가 정해준다**면 **Props** 쪽을 먼저 의심하세요. 예외는 나중에 배우면 됩니다."

**전환**

> "용어 정의를 슬라이드로 다시 정리할게요."

---

## 슬라이드: 4.1 Props란?

**멘트**

> "**Props**는 **Properties**의 줄임말입니다."

> "**부모 컴포넌트가 자식 컴포넌트에 데이터를 넘기는 방법**이에요."

> "**특징: 읽기 전용, immutable** — 자식 입장에서는 **props를 직접 수정하지 않는다**고 기억하세요. 바꿔야 하면 **부모의 state를 바꾸거나**, **콜백 함수**를 props로 받는 패턴을 씁니다. 오늘은 **읽기만 한다**에 집중합니다."

**전환**

> "Props와 State를 **한 슬라이드에서 대비**해 볼게요."

---

## 슬라이드: 4.1 Props vs State (표 형 비교)

**멘트**

> "**Props** — **부모에서 전달**, 자식은 **변경 불가**, **밖에서 들어온 데이터** 같은 느낌."

> "**State** — **컴포넌트 내부**에서 관리, **바꾸면 리렌더링**, **안에서 움직이는 데이터** 같은 느낌."

**전환**

> "코드로 **Props 사용법**을 보겠습니다."

---

## 슬라이드: 4.2 Props 사용하기

**멘트**

> "부모 `App`에서 `<Greeting name="John" age={25} />` 처럼 씁니다. **`name`은 문자열**이라 따옴표로, **`age`는 숫자**라서 **중괄호**로 넘깁니다. 문자열로 넘기면 숫자가 아니라 문자가 될 수 있어서, 숫자는 **중괄호**로 주는 습관이 좋아요."

> "자식 `Greeting`에서 `function Greeting(props)` 로 받아서 **`props.name`**, **`props.age`** 로 읽습니다."

**전환**

> "매번 `props.` 를 쓰기 귀찮으니 **구조 분해**로 깔끔하게 씁니다."

## 슬라이드: 4.3 구조 분해 할당

**멘트**

> "`function Greeting({ name, age })` — **매개변수에서 바로 구조 분해**합니다. 안에서는 `name`, `age`만 쓰면 돼요."

> "구조 분해는 자바스크립트 문법이라, React만의 문법은 아니에요. **객체에서 필요한 속성만 꺼내는 것**입니다."

**전환**

> "부모가 준 값만으로는 **클릭할 때마다 바뀌는 숫자** 같은 걸 만들기 어렵습니다. **안에서 바뀌는 데이터**가 **State**입니다. **5부**로 갑니다."

## 슬라이드: 5. State (90분) — 섹션 구분

**멘트**

> "State와 **useState** 훅이 오늘의 **핵심 중 하나**예요. **퀴즈 정답**이기도 했죠 — **바뀌면 화면이 다시 그려지는 내부 데이터**가 **state**입니다."

---

## 슬라이드: 5.1 State란?

**멘트**

> "**State**는 **컴포넌트 내부에서 바뀔 수 있는 데이터**입니다."

> "**특징**: State가 바뀌면 React는 **그 컴포넌트를 다시 렌더링**합니다. 화면이 최신 데이터와 맞게 갱신되는 거예요."

> "**관리 방법**: 요즘 함수형 컴포넌트에서는 **`useState` 훅**을 씁니다. **Hook**은 '갈고리'라는 뜻인데, React가 제공하는 **특별한 함수**로 **state를 걸어 두는 것**이라고 이해하시면 됩니다."

**전환**

> "`useState` 문법을 슬라이드 예제로 보겠습니다."

---

## 슬라이드: 5.2 useState Hook

**멘트**

> "맨 위에 `import { useState } from 'react';` — React 패키지에서 **useState만** 가져옵니다."

> "`const [count, setCount] = useState(0);` — **배열 구조 분해**예요.  
> **첫 번째** `count`는 **현재 값**,  
> **두 번째** `setCount`는 **값을 바꿀 때 쓰는 함수**,  
> **`useState(0)`** 의 **0**은 **초기값**입니다."

> "버튼에서는 `onClick={() => setCount(count + 1)}` 처럼 **setter를 호출**합니다. **count에 직접 대입하면 안 되고**, 반드시 **setCount**로 바꿔야 React가 변경을 알고 화면을 갱신해요."

**전환**

> "setter를 쓸 때 **함수형 업데이트**라는 게 있습니다. 조금 어렵지만 실무에서 중요해요."

---

## 슬라이드: 5.3 State 업데이트 - 핵심 개념

**멘트**

> "`setCount(prev => prev + 1)` 형태를 **함수형 업데이트**라고 합니다. **이전 값**을 인자로 받아서 **새 값을 계산**해요."

> "React의 state 업데이트는 상황에 따라 **비동기적으로 묶일 수 있다**고들 합니다. 그래서 **`setCount(count + 1)`** 를 **연속으로 여러 번** 호출하면, 기대와 다르게 동작할 수 있어요. **항상 최신 이전 값**에 기대고 싶으면 **`prev => prev + 1`** 이 안전합니다."

> "초보: **한 번 클릭에 한 번씩만** 올리면 `count + 1` 도 대부분 괜찮습니다. **연속 클릭·연속 업데이트**를 다룰 때 **함수형 업데이트**를 습관화하세요."

**전환**

> "코드 예시 슬라이드로 같은 내용을 한 번 더 볼게요."

---

## 슬라이드: 5.3 State 업데이트 - 코드 예시

**멘트**

> "`handleIncrease`에서 `setCount(prevCount => prevCount + 1)` — **권장 패턴**입니다."

> "같은 슬라이드에 **직접 증가** 예시도 있어요. 설명용으로만 이해하고, **이전 값에 의존할 때는 함수형**을 쓰세요."

**전환**

> "입력 폼처럼 **여러 값**을 동시에 다루는 예시입니다."

---

## 슬라이드: 5.4 여러 State 사용

**멘트**

> "`name`, `email`, `age`처럼 **useState를 여러 번** 쓸 수 있어요."

> "입력창에서는 **`value={name}`** 과 **`onChange={(e) => setName(e.target.value)}`** 를 같이 씁니다. 이렇게 하면 **입력 상자가 state와 동기화**됩니다. **제어 컴포넌트**라고 부릅니다. 처음엔 패턴으로 외워도 됩니다."

> "**e.target.value**는 **지금 입력창에 적힌 문자열**이에요."

**전환**

> "사용자가 클릭하고, 입력하고, 폼을 제출하는 건 **이벤트**로 처리합니다. **6부**예요."
